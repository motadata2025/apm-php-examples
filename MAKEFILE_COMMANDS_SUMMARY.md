# APM PHP Examples - Makefile Commands Summary

## 📋 Overview

This document provides comprehensive terminal-level documentation for every Makefile command across all PHP framework applications in the APM PHP Examples project. Each command is documented with exact terminal usage, interactive prompts, visible output, and troubleshooting information.

## 🎯 Core Philosophy

**Always use `make` commands** - Never call scripts directly. The Makefiles provide consistent interfaces, proper validation, and reliable operation across all applications.

## 📑 Quick Index

### Common Commands (All Applications)
- `make help` - Show available commands
- `make setup` - System requirements validation and Docker service startup
- `make compile` - Interactive application configuration (PHP version, deployment type)
- `make start` - Deploy and start the application
- `make status` - Comprehensive application status report
- `make stop` - Stop application and cleanup
- `make down` - Complete cleanup and reset

### Extended Commands (Simple PHP, Symfony)
- `make php-status` - PHP version diagnostics and compatibility
- `make network-status` - Network configuration and IP change detection

### Framework-Specific Commands
- **Laravel**: `make artisan`, `make dev`
- **CodeIgniter**: `make spark`, `make dev`
- **Slim Framework**: `make dev`, `make update`

### Development Commands (All Applications)
- `make test` - Run framework-specific tests
- `make install` - Install dependencies with Composer
- `make update` - Update dependencies (Laravel, CodeIgniter, Slim)

## 🔧 Global Notes

### Docker Compose Detection
All applications automatically detect and use:
- `docker compose` (v2) if available
- `docker-compose` (v1) as fallback
- Prints detection result: "Using docker compose v2" or "Using docker-compose v1"

### Configuration Management
- **Primary Config**: `config/app.env` (generated by `make compile`)
- **Template**: `config/app.env.example` (version controlled)
- **Single Source**: All services read from `config/app.env`

### simple-php — `make setup`

**Purpose (one line):**  
Validate system requirements, start Docker services (MySQL, PostgreSQL, Redis), and ensure all services are healthy.

**Terminal usage:**  
- `make setup`

**Interactive choices / options shown at runtime:**  
- **PHP Extension Installation**: If missing extensions detected:
  - "Install missing PHP extensions? (1) All versions (2) Select versions (3) Skip"
  - If option 2 selected: "Select PHP versions (space-separated, e.g., 8.3 8.4):"
  - 30-second timeout for user input

**Visible terminal output (what you will see, step-by-step):**  
1. "🔍 Checking system requirements..." (blue)
2. "🔧 Fixing script permissions..." with chmod operations
3. "🐘 Checking PHP versions..." with version detection results
4. "📦 Checking PHP extensions..." with extension status
5. "🌐 Checking web servers..." (Apache, Nginx, PHP-FPM availability)
6. "🐳 Checking Docker..." with daemon status
7. "🚀 Starting Docker services..." with container startup output
8. "⏳ Waiting for services to be ready..." with health check progress
9. Final summary: "✅ System is ready for Simple PHP deployment"

**High-level step-by-step flow (what the Makefile does — conceptually):**  
1. Fix script permissions (chmod +x scripts/*)
2. Detect available PHP versions (8.1-8.4)
3. Check required PHP extensions (mysql, pgsql, redis)
4. Validate web server availability (Apache, Nginx, PHP-FPM)
5. Verify Docker daemon is running
6. Start Docker containers using docker-compose.yml
7. Wait for container health checks to pass
8. Print system readiness summary

**Preconditions (things that must be true before running):**  
- Docker daemon running and accessible
- `docker` and `docker compose` (or `docker-compose`) binaries available

**Postconditions (state after success):**  
- Containers running: `simple_php_mysql`, `simple_php_postgres`, `simple_php_redis`
- Port mappings: MySQL:3307, PostgreSQL:5433, Redis:6380
- All containers show "healthy" status

**Success criteria (how to programmatically verify success):**  
- `docker ps` shows 3 containers with "healthy" status
- `docker compose ps` shows all services "Up"
- Ports accessible: `telnet localhost 3307`, `telnet localhost 5433`, `telnet localhost 6380`

**Failure modes & debugging steps (practical, terminal-first):**  
- **Docker not running**: 
  - Symptom: "Cannot connect to the Docker daemon"
  - Fix: `sudo systemctl start docker` or start Docker Desktop
- **Port conflicts**:
  - Symptom: "Port already in use" during container startup
  - Debug: `ss -tuln | grep 3307` to check port usage
  - Fix: Stop conflicting services or change ports in `config/app.env`
- **Container health check failures**:
  - Symptom: Container stuck in "starting" state
  - Debug: `docker compose logs simple_php_mysql`
  - Fix: `docker compose down -v && make setup` (remove volumes and restart)

**Env variables & config values that affect this command:**  
- `MYSQL_PORT`, `POSTGRES_PORT`, `REDIS_PORT` from `config/app.env` (if exists)
- `DOCKER_BUILDKIT` affects Docker build behavior
- `COMPOSE_PROJECT_NAME` affects container naming

**Compatibility / edge cases to note:**
- Docker Compose v1 vs v2: Automatically detects and uses appropriate command
- Missing PHP versions: Continues with available versions, warns about missing ones
- Network conflicts: Automatically assigns alternative subnets if conflicts detected

   - User enters: `8.4` (or desired version)

2. **Network Configuration**:
   - "Select network interface: (1) localhost (2) public IP (3) all interfaces"
   - Shows current IP and interface options

3. **Deployment Type Selection**:
   - "Available deployment options for PHP X.X:"
   - Lists: `php-cli`, `apache-mod-php`, `apache-fpm`, `nginx-fpm`
   - Shows compatibility status for each option

4. **Dependency Installation**:
   - "Install/update dependencies? (1) Install (2) Update (3) Skip"

**Visible terminal output (what you will see, step-by-step):**
1. "🔨 Compiling Simple PHP application..." (blue)
2. "🔍 Checking application state..." (ensures not running)
3. "🌐 Network Configuration" with IP detection and interface selection
4. "🐘 PHP Version Configuration" with version status and selection
5. "📦 Dependency Management" with Composer operations
6. "🌐 Web Server Selection" with deployment options
7. "💾 Saving configuration..." with config file generation
8. "📋 Configuration Summary" showing final settings

**High-level step-by-step flow (what the Makefile does — conceptually):**
1. Verify application is not currently running
2. Detect network interfaces and current IP
3. Present PHP version options and validate selection
4. Run Composer install/update based on user choice
5. Show deployment options compatible with selected PHP version
6. Generate `config/app.env` with all selected settings
7. Validate configuration consistency
8. Display final configuration summary

**Preconditions (things that must be true before running):**
- Application not currently running (`make stop` if needed)
- `composer` available in PATH
- PHP versions available on system
- Docker services running (from `make setup`)

**Postconditions (state after success):**
- `config/app.env` file created with complete configuration
- Composer dependencies installed for selected PHP version
- Application ready for `make start`

**Success criteria (how to programmatically verify success):**
- `config/app.env` exists and contains valid configuration
- `composer show` lists installed packages
- Configuration values match user selections

**Failure modes & debugging steps (practical, terminal-first):**
- **Composer fails on platform requirements**:
  - Symptom: "Your requirements could not be resolved to an installable set of packages"
  - Debug: `composer why-not php ^8.4` to see conflicts
  - Fix: Install missing PHP extensions or use `composer install --ignore-platform-reqs`
- **PHP version not available**:
  - Symptom: "PHP X.X not found on system"
  - Debug: `php8.4 --version` to verify installation
  - Fix: Install PHP version with `sudo apt install php8.4-cli php8.4-fpm`
- **Web server not available**:
  - Symptom: "Apache/Nginx not available for deployment"
  - Debug: `systemctl status apache2` or `systemctl status nginx`
  - Fix: Install web server or select different deployment type

**Env variables & config values that affect this command:**
- `PHP_VERSION` (overridden by user selection)
- `COMPOSER_MEMORY_LIMIT` affects Composer operations
- `APP_ENV` influences Composer flags (--no-dev for production)

**Compatibility / edge cases to note:**
- Composer platform.php: Sets PHP version constraint based on selection
- Extension requirements: Different frameworks require different PHP extensions
- Memory limits: Large frameworks may need increased Composer memory limit

### simple-php — `make start`

**Purpose (one line):**
Deploy and start the application using configuration from `make compile`.

**Terminal usage:**
- `make start`

**Interactive choices / options shown at runtime:**
None (uses configuration from `config/app.env`).

**Visible terminal output (what you will see, step-by-step):**
1. "🚀 Starting Simple PHP application with isolated containers..." (blue)
2. "📋 Loading configuration from config/app.env"
3. "🔍 Validating Docker services..." with health checks
4. "🌐 Network validation..." with IP change detection
5. "🚀 Deploying application..." with deployment-specific steps
6. "✅ Application started successfully"
7. Access information with URLs and management commands

**High-level step-by-step flow (what the Makefile does — conceptually):**
1. Load configuration from `config/app.env`
2. Validate Docker containers are healthy
3. Check for network changes and update config if needed
4. Execute deployment based on selected type:
   - **php-cli**: Start built-in PHP server
   - **apache-fpm**: Configure virtual host and restart Apache
   - **nginx-fpm**: Configure server block and restart Nginx
   - **apache-mod-php**: Configure mod_php and restart Apache
5. Create PID files for process tracking
6. Validate application responds to HTTP requests
7. Display access information and management commands

**Preconditions (things that must be true before running):**
- `config/app.env` exists (run `make compile` first)
- Docker services healthy (run `make setup` if needed)
- Selected web server available and configured
- Ports available (no conflicts)

**Postconditions (state after success):**
- Application accessible on configured port
- Process running and tracked via PID file
- Web server configured (if applicable)
- Application logs being written

**Success criteria (how to programmatically verify success):**
- HTTP request to application returns 200 status
- `curl -s http://localhost:8000/health` returns JSON response
- Process PID file exists and process is running

**Failure modes & debugging steps (practical, terminal-first):**
- **Port already in use**:
  - Symptom: "Address already in use" or "Port 8000 is busy"
  - Debug: `ss -tuln | grep 8000` to find conflicting process
  - Fix: `sudo lsof -ti:8000 | xargs sudo kill` or change port in config
- **Web server configuration error**:
  - Symptom: "Apache/Nginx configuration failed"
  - Debug: `sudo apache2ctl configtest` or `sudo nginx -t`
  - Fix: Check syntax errors in generated config files
- **Application startup failure**:
  - Symptom: Process starts but HTTP requests fail
  - Debug: Check logs in `logs/application.log`
  - Fix: Verify database connections and dependencies

**Env variables & config values that affect this command:**
All values from `config/app.env`:
- `APP_PORT`, `NETWORK_INTERFACE` for binding
- `PHP_VERSION`, `DEPLOYMENT_TYPE` for execution method
- `MYSQL_PORT`, `POSTGRES_PORT`, `REDIS_PORT` for database connections

**Compatibility / edge cases to note:**
- IP changes: Automatically detects and updates configuration
- Firewall rules: Creates UFW rules for public access if needed
- Process management: Different PID handling per deployment type

### simple-php — `make down`

Complete cleanup removing all containers, volumes, configuration files, and application data.

**Terminal usage:**
- `make down`

**Interactive choices / options shown at runtime:**
None (complete cleanup, no confirmation prompt).

**Visible terminal output (what you will see, step-by-step):**

  - Symptom: "Error removing containers"

## 🚨 Troubleshooting Appendix

### Common Docker Issues

#### **Container Health Check Failures**
**Symptoms:**
- Container stuck in "starting" state
- `docker ps` shows "health: starting" for extended time
- Application cannot connect to database

**Debugging Steps:**
```bash
# Check container logs
docker compose logs <service_name>

# Check container health
docker inspect <container_name> | grep -A 10 Health

# Test port connectivity
telnet localhost <port>
```

**Resolution:**
```bash
# Remove volumes and restart
docker compose down -v
make setup

# If persistent, check for port conflicts
ss -tuln | grep <port>
```

#### **Port Binding Conflicts**
**Symptoms:**
- "Port already in use" during container startup
- "Address already in use" errors
- Containers fail to start

**Debugging Steps:**
```bash
# Find process using port
sudo lsof -i :<port>

# Check all listening ports
ss -tuln | grep <port>

# Check Docker port mappings
docker port <container_name>
```

**Resolution:**
```bash
# Kill conflicting process
sudo kill -9 <PID>

# Or change port in config/app.env
nano config/app.env
make stop && make start
```

### Common Composer Issues

#### **Platform Requirements Not Met**
**Symptoms:**
- "Your requirements could not be resolved to an installable set of packages"
- "requires php ^8.4 but your php version is 8.3"
- Extension missing errors

**Debugging Steps:**
```bash
# Check PHP version
php --version

# Check installed extensions
php -m | grep -E "(mysql|pgsql|redis)"

# Diagnose Composer issues
composer diagnose
composer why-not php ^8.4
```

**Resolution:**
```bash
# Install missing PHP version
sudo apt install php8.4-cli php8.4-fpm php8.4-mysql php8.4-pgsql

# Install missing extensions
sudo apt install php8.4-redis php8.4-mbstring php8.4-xml

# Temporary workaround
composer install --ignore-platform-reqs
```

#### **Memory Limit Issues**
**Symptoms:**
- "Fatal error: Allowed memory size exhausted"
- Composer hangs during installation
- Large framework installations fail

**Debugging Steps:**
```bash
# Check current memory limit
php -ini | grep memory_limit

# Check Composer memory usage
composer install -vvv
```

**Resolution:**
```bash
# Increase memory limit temporarily
COMPOSER_MEMORY_LIMIT=-1 composer install

# Or set permanently
export COMPOSER_MEMORY_LIMIT=2G
```

### Common Network Issues

#### **Application Not Accessible**
**Symptoms:**
- "Connection refused" when accessing application
- Timeout errors from browser
- curl fails to connect

**Debugging Steps:**
```bash
# Check if process is running
ps aux | grep php

# Check port binding
ss -tuln | grep <app_port>

# Test local connectivity
curl -v http://localhost:<port>

# Check firewall rules
sudo ufw status
```

**Resolution:**
```bash
# Restart application
make stop && make start

# Check configuration
cat config/app.env

# Open firewall port (if needed)
sudo ufw allow <port>
```

#### **IP Address Changes**
**Symptoms:**
- Application accessible locally but not externally
- "Network unreachable" errors
- Configuration shows old IP addresses

**Debugging Steps:**
```bash
# Check current IP
ip addr show

# Check network status
make network-status

# Verify interface binding
ss -tuln | grep <port>
```

**Resolution:**
```bash
# Reconfigure for new network
make compile

# Or update network settings
make stop && make start
```

### Common Web Server Issues

#### **Apache/Nginx Configuration Errors**
**Symptoms:**
- "Configuration test failed"
- Web server won't start after deployment
- 404 errors for application

**Debugging Steps:**
```bash
# Test Apache configuration
sudo apache2ctl configtest

# Test Nginx configuration
sudo nginx -t

# Check web server status
systemctl status apache2
systemctl status nginx
```

**Resolution:**
```bash
# Fix configuration syntax
sudo nano /etc/apache2/sites-available/<app>.conf

# Restart web server
sudo systemctl restart apache2

# Or disable and re-enable site
sudo a2dissite <app>
sudo a2ensite <app>
sudo systemctl reload apache2
```

### Emergency Recovery Procedures

#### **Complete Reset**
When all else fails, perform a complete reset:
```bash
# Stop everything
make down

# Clean Docker system
docker system prune -af
docker volume prune -f

# Reset application
rm -f config/app.env
rm -rf logs/

# Start fresh
make setup
make compile
make start
```

#### **Backup and Restore Configuration**
```bash
# Backup working configuration
cp config/app.env config/app.env.backup

# Restore from backup
cp config/app.env.backup config/app.env
make stop && make start
```

## 🎯 Best Practices

### Development Workflow
1. **Initial Setup**: `make setup` → `make compile` → `make start`
2. **Daily Development**: `make status` to check health
3. **Configuration Changes**: `make compile` → `make stop` → `make start`
4. **Testing**: `make test` before committing changes
5. **Cleanup**: `make stop` when done, `make down` for complete reset

### Production Deployment
1. **System Validation**: `make setup` on production server
2. **Production Configuration**: `make compile` with production settings
3. **Deployment**: `make start` with monitoring
4. **Health Monitoring**: Regular `make status` checks
5. **Log Monitoring**: Check `logs/application.log` regularly

### Troubleshooting Strategy
1. **Check Status**: `make status` for overview
2. **Check Logs**: Application and Docker container logs
3. **Validate Configuration**: Verify `config/app.env` settings
4. **Test Connectivity**: Port accessibility and network configuration
5. **Progressive Reset**: `make stop` → `make start` → `make down` → full reset
